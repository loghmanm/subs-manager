# سامانه مدیریت مشترکین — راهنمای استقرار روی میکروتیک (Container)

این برنامه یک اپلیکیشن کامل Node.js است (رابط کاربری فارسی، دیتابیس SQLite داخلی، زمان‌بند ارسال پیام تلگرام)
که به‌صورت **Container** روی خود روتر میکروتیک شما (RouterOS 7.23، x86_64 / CHR) اجرا می‌شود.

---

## ۱) پیش‌نیازها

- روتر شما: RouterOS 7.23 روی x86_64 (VMware CHR) — **از قابلیت Container پشتیبانی می‌کند.** ✅
- یک کامپیوتر دیگر (ویندوز/مک/لینوکس) که **Docker** روی آن نصب باشد، فقط برای «ساختن» ایمیج برنامه.
  (خود میکروتیک نمی‌تواند ایمیج بسازد؛ فقط می‌تواند ایمیج آماده را اجرا کند.)
- دسترسی WinBox یا FTP به روتر برای آپلود فایل.

---

## ۲) ساخت ایمیج Docker (روی کامپیوتر جانبی)

۱. پوشه‌ی `subs-app` (که در پیوست این پیام است) را باز کنید. یک فایل `.env` در ریشه‌ی پروژه بسازید
   (از روی `.env.example` کپی کنید) و اطلاعات تلگرام/رمز عبور را در آن بررسی و در صورت نیاز تغییر دهید.

۲. در همان پوشه این دستورات را اجرا کنید:

```bash
docker build -t subs-manager:latest .
docker save subs-manager:latest -o subs-manager.tar
```

فایل `subs-manager.tar` (حدود ۸۰-۱۲۰ مگابایت) ساخته می‌شود. همین فایل را در مرحله بعد به روتر آپلود می‌کنیم.

> **نکته:** اگر یک سرور لینوکسی یا VPS با دسترسی اینترنت در اختیار دارید، می‌توانید به‌جای این روش،
> ایمیج را در یک حساب رایگان Docker Hub push کنید و در مرحله بعد به‌جای آپلود فایل tar،
> مستقیماً با `remote-image=` آدرس آن را به روتر بدهید تا خودش دانلود کند. برای سادگی، این راهنما روش آپلود مستقیم فایل را توضیح می‌دهد.

---

## ۳) فعال‌سازی Container Mode روی روتر

به‌صورت پیش‌فرض قابلیت Container در RouterOS غیرفعال است. در ترمینال روتر:

```
/system/device-mode/update container=yes
```

سیستم از شما می‌خواهد تغییر را با **ریبوت سرد** (Cold Reboot / خاموش‌روشن کردن ماشین مجازی) تأیید کنید،
چون دکمه Reset فیزیکی روی CHR وجود ندارد. VM را از پنل VMware/هایپروایزر خودتان ریست/ریستارت کنید،
نه از طریق دستور reboot معمولی RouterOS.

بعد از بالا آمدن دوباره، بررسی کنید:

```
/system/device-mode/print
```

باید `container: yes` را ببینید.

---

## ۴) بررسی فضای ذخیره‌سازی

```
/disk/print
```

- اگر یک ردیف دیسک (مثلاً `disk1`) دیدید، مسیرهای زیر را با پیشوند `disk1/` استفاده کنید.
- اگر هیچ ردیفی نبود (معمول در CHR که کل دیسک مجازی یک پارتیشن است)، پیشوند دیسک لازم نیست
  و می‌توانید مستقیماً از مسیرهایی مثل `containers/subs-manager` استفاده کنید.
  در ادامه‌ی این راهنما فرض می‌کنیم پیشوند دیسک لازم نیست؛ در غیر این صورت `disk1/` را به ابتدای هر `root-dir` و `src` اضافه کنید.

با توجه به گزارش `/system/resource/print` شما (حدود ۵.۸ گیگابایت فضای آزاد)، جای کافی برای این برنامه (کمتر از ۲۰۰ مگابایت) وجود دارد.

---

## ۵) ساخت شبکه‌ی Container (veth + bridge + NAT)

```
/interface/bridge/add name=bridge-containers
/ip/address/add address=172.30.0.1/24 interface=bridge-containers

/interface/veth/add name=veth-subs address=172.30.0.2/24 gateway=172.30.0.1
/interface/bridge/port/add bridge=bridge-containers interface=veth-subs

/ip/firewall/nat/add chain=srcnat action=masquerade src-address=172.30.0.0/24 comment="Container NAT"
```

سپس یک قانون Port-Forward برای دسترسی به پنل از داخل شبکه‌ی خودتان (روی پورت دلخواه، مثلاً ۸۰۹۵ از IP خود روتر):

```
/ip/firewall/nat/add chain=dstnat action=dst-nat protocol=tcp dst-port=8095 to-addresses=172.30.0.2 to-ports=8080 comment="Subs Manager Panel"
```

> **امنیت:** چون این پنل شامل اطلاعات مشترکین است، بهتر است دسترسی به پورت ۸۰۹۵ را فقط برای IPهای مدیریتی خودتان مجاز کنید. مثال محدودسازی به یک IP یا رنج مشخص:
> ```
> /ip/firewall/filter/add chain=input protocol=tcp dst-port=8095 src-address=YOUR_ADMIN_IP action=accept place-before=0
> /ip/firewall/filter/add chain=input protocol=tcp dst-port=8095 action=drop
> ```
> به‌جای `YOUR_ADMIN_IP` رنج شبکه داخلی یا IP کامپیوتر خودتان را قرار دهید.

---

## ۶) آپلود فایل ایمیج به روتر

با **WinBox → Files** (یا FTP)، فایل `subs-manager.tar` که در مرحله ۲ ساختید را در ریشه فایل‌سیستم روتر آپلود کنید (Drag & Drop کافی است).

---

## ۷) ساخت و اجرای Container

```
/container/add file=subs-manager.tar interface=veth-subs root-dir=containers/subs-manager name=subs-manager start-on-boot=yes logging=yes
```

اگر در مرحله ۴ متوجه شدید باید از `disk1` استفاده کنید، دستور را این‌طور بنویسید:

```
/container/add file=disk1/subs-manager.tar interface=veth-subs root-dir=disk1/containers/subs-manager name=subs-manager start-on-boot=yes logging=yes
```

بعد از اجرای دستور، RouterOS شروع به «extract» کردن ایمیج می‌کند (ممکن است چند دقیقه طول بکشد). وضعیت را بررسی کنید:

```
/container/print detail
```

وقتی `status` به `stopped` تغییر کرد یعنی آماده اجراست:

```
/container/start [find name=subs-manager]
```

دوباره `/container/print detail` بزنید؛ باید `status: running` را ببینید.

### مشاهده لاگ‌های برنامه (برای عیب‌یابی)

```
/log/print where topics~"container"
```

## ۸) ورود به پنل

از داخل شبکه‌ی خودتان، در مرورگر باز کنید:

```
http://IP_روتر_شما:8095
```

نام کاربری: **loghman**
رمز عبور: همانی که در فایل `.env` تعیین کردید (پیش‌فرض پیشنهادی: `MAXima6767` — پیشنهاد می‌شود بعداً عوض کنید).

داده‌های قدیمی هر دو فایل شما (۳۶۸ مشترک) در اولین اجرا به‌صورت خودکار وارد پایگاه‌داده می‌شوند.
چون بسیاری از رکوردهای فایل «Expired_Disabled» قدیمی و منقضی هستند، به‌صورت طبیعی با وضعیت «منقضی شده» نمایش داده می‌شوند؛ کافی است هر مشترکی که هنوز فعال است را ویرایش/تاریخ‌گذاری کنید یا موارد غیرلازم را حذف کنید.

---

## ۹) نگه‌داری اطلاعات (Data Persistence)

دیتابیس مشترکین در مسیر `/app/data/subscribers.db` **داخل** کانتینر ذخیره می‌شود، اما چون این مسیر یک `VOLUME` تعریف‌شده در Dockerfile است،
RouterOS آن را به‌صورت خودکار در پوشه‌ی `root-dir` شما (همان `containers/subs-manager`) نگه می‌دارد — یعنی حتی اگر کانتینر را
Stop/Start یا Remove/Add-دوباره کنید، اطلاعات مشترکین از بین نمی‌رود (تا زمانی که خود پوشه `root-dir` را پاک نکنید).

**پیشنهاد:** هر چند وقت یک‌بار از پوشه‌ی `containers/subs-manager` (از طریق WinBox → Files) یک نسخه پشتیبان بگیرید.

---

## ۱۰) به‌روزرسانی برنامه در آینده

اگر تغییری در کد دادید و ایمیج جدید ساختید:

```
/container/stop [find name=subs-manager]
/container/remove [find name=subs-manager]
```

سپس فایل tar جدید را آپلود و دوباره دستور `/container/add ...` را با همان `root-dir` قبلی اجرا کنید — دیتابیس شما دست‌نخورده باقی می‌ماند.

---

## ۱۱) متغیرهای محیطی مهم (در فایل `.env` قبل از build)

| متغیر | توضیح |
|---|---|
| `APP_USERNAME` / `APP_PASSWORD` | نام کاربری و رمز ورود به پنل |
| `TELEGRAM_BOT_TOKEN` / `TELEGRAM_CHAT_ID` | مشخصات ربات تلگرام برای ارسال اعلان‌ها |
| `CRON_SCHEDULE` | زمان اجرای روزانه‌ی بررسی انقضا (فرمت cron، پیش‌فرض `0 9 * * *` یعنی هر روز ساعت ۹ صبح) |
| `SESSION_SECRET` | یک رشته‌ی تصادفی دلخواه برای امنیت نشست ورود |

اگر خواستید بعداً این مقادیر را بدون rebuild تغییر دهید، می‌توانید هنگام ساخت کانتینر با پارامتر `envlist=` یک لیست Environment Variable در RouterOS تعریف و به کانتینر متصل کنید (مستندات: `/container/envs/add`).

---

## ۱۲) منطق اعلان‌های تلگرام (یادآوری خودکار)

- هر مشترکی که با دکمه‌ی «افزودن به لیست نظارت» (آیکون ✓ کنار هر ردیف) انتخاب شود، وارد چرخه‌ی بررسی روزانه می‌شود.
- هر روز رأس ساعت تنظیم‌شده (پیش‌فرض ۹ صبح):
  - **۵ روز مانده** به انقضا → یک پیام یادآوری
  - **۳ روز مانده** → یک پیام هشدار
  - **روز انقضا** → یک پیام «اشتراک منقضی شد»
- هر پیام فقط یک‌بار برای هر رویداد ارسال می‌شود (رکورد ارسال در دیتابیس ثبت می‌شود).
- اگر تاریخ انقضای یک مشترک را ویرایش/تمدید کنید، چرخه‌ی اعلان‌ها برای او خودکار ریست می‌شود.
- دکمه‌ی «تست اتصال تلگرام» در بالای صفحه برای اطمینان از صحت توکن/چت‌آی‌دی در هر زمان قابل استفاده است.

---

## محدودیت‌های شناخته‌شده

- وب‌سرور داخلی خود میکروتیک (بدون Container) قابلیت اجرای این نوع برنامه را ندارد؛ به همین علت از Container استفاده شد.
- این پنل از HTTP ساده استفاده می‌کند (بدون HTTPS)؛ چون فقط از شبکه داخلی/با فایروال محدود به آن دسترسی دارید، ریسک آن کنترل‌شده است. اگر بخواهید از بیرون شبکه هم به آن دسترسی داشته باشید، پیشنهاد می‌شود از یک VPN (مثلاً WireGuard که خود میکروتیک هم پشتیبانی می‌کند) به شبکه داخلی وصل شوید، به‌جای باز کردن این پورت مستقیم روی اینترنت.
